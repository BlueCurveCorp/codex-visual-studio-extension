# CFP-001: Slash Commands

Priority: P0  
Status: Todo  
Owner: AI agent  
Depends on: none

## Goal

Add first-class slash command support to the Visual Studio Codex composer so users can control common Codex actions without leaving the chat input.

Initial command scope:

- `/status`
- `/review`
- `/goal`
- `/plan`
- `/mcp`
- `/cloud`
- `/local`
- `/auto-context`
- `/feedback`
- `$skill` invocation and autocomplete

## User Value

Slash commands make the extension feel like a native Codex surface instead of a plain chat wrapper. They also provide discoverable access to features already partly present in the extension, including status, Plan mode, skills, MCP, and rate-limit information.

## Functional Requirements

1. Detect slash command mode when the composer starts with `/`.
2. Detect skill invocation mode when the current token starts with `$`.
3. Show a themed suggestion popup/list near the composer.
4. Filter commands by typed prefix.
5. Support keyboard navigation:
   - Up/down moves selection.
   - Enter accepts selected command.
   - Escape closes suggestions.
   - Tab accepts selected command when suggestions are open.
6. Support mouse click selection.
7. Keep existing `@file` mention behavior working.
8. Do not send slash command text as a normal prompt unless the command explicitly maps to a prompt.
9. Commands that map to local UI actions should run locally.
10. Commands that map to Codex CLI behavior should use the existing process-service execution path when possible.

## Command Behavior

### `/status`

Show a local status summary in the thread:

- Codex CLI path and readiness
- account/auth state when available
- selected model
- reasoning effort
- verbosity
- approval policy
- sandbox mode
- current thread ID if known
- context usage estimate
- rate-limit summary
- MCP server count and health summary
- enabled skill count

Implementation notes:

- Prefer an internal status card/message over launching a Codex process.
- Reuse existing `CodexRateLimitSummary`, `CodexEnvironmentStatus`, `McpServers`, `Skills`, and context estimate properties.

### `/review`

For CFP-001, provide a thin entry point:

- If CFP-002 is not implemented yet, insert a prompt such as `Review the current uncommitted changes. Focus on bugs, regressions, and missing tests.`
- If CFP-002 is implemented, open the Review/Diff pane and set review mode.

Implementation must avoid blocking CFP-001 on CFP-002.

### `/goal`

MVP behavior:

- If goal support is available in the local Codex CLI, pass through to Codex using the supported command or prompt syntax.
- If not available, open a local dialog or inline prompt to capture a durable goal statement and prepend it to subsequent prompts as thread context.

Settings:

- Add persisted `GoalsEnabled` only if the extension needs a local fallback.
- Do not write `~/.codex/config.toml` without explicit user action.

### `/plan`

Toggle existing Plan mode if present. If current mode concepts differ, map `/plan` to the extension's current plan/normal mode selector.

### `/mcp`

Open the existing MCP settings/status section and refresh integrations.

### `/cloud`

For CFP-001, expose only a stub that routes to CFP-004:

- If cloud support is not implemented, show a localized message that cloud delegation is planned and link to CFP-004 behavior.
- If CFP-004 is implemented, switch current run mode to Cloud.

### `/local`

Set current run mode to local execution. Initially this can mean the current default local CLI execution path.

### `/auto-context`

Toggle the setting introduced by CFP-003. Before CFP-003 exists, show a localized message that Auto Context is not installed yet.

### `/feedback`

Open a Codex feedback URL or app deep link if available. Otherwise open Codex docs/support link in the browser.

### `$skill`

Support two flows:

- Typing `$` shows enabled skills.
- Selecting a skill inserts `$skill-name` into the composer.

Reuse existing `Skills`, `RemoteSkills`, `DetectedPromptSkills`, and skill enablement behavior.

## Non-Functional Requirements

- UI must use Visual Studio theme resources.
- All strings must be localized.
- The suggestion list must not overlap composer controls at narrow widths.
- The feature must be testable without network access.
- Missing CLI support must degrade gracefully.

## Implementation Steps

1. Read `AGENTS.md`, this file, and current Codex docs for slash commands.
2. Use `search_graph` to find composer text, prompt mention, skill, and command handling.
3. Add a model such as `ComposerCommandSuggestion`.
4. Add a service such as `ComposerCommandService` for command metadata and filtering.
5. Add view-model state:
   - `ComposerSuggestions`
   - `IsComposerSuggestionOpen`
   - `SelectedComposerSuggestion`
   - `ComposerSuggestionMode`
6. Extend prompt text change handling to detect `/`, `$`, and existing `@`.
7. Add XAML suggestion UI using existing styles and theme resources.
8. Add keyboard handling in `CodexToolWindowControl.xaml.cs`.
9. Implement command dispatch in `CodexToolWindowViewModel`.
10. Add localization keys for command labels, descriptions, unavailable messages, and status output.
11. Ensure slash commands are excluded from normal prompt send unless intended.
12. Build and fix analyzer warnings.

## Acceptance Criteria

- Typing `/` opens command suggestions.
- Typing `$` opens skill suggestions.
- `/status` posts a useful status summary without starting a normal task.
- `/plan` toggles Plan mode.
- `/mcp` opens MCP settings/status.
- `/review`, `/cloud`, and `/auto-context` degrade gracefully when their later tasks are not implemented.
- Existing `@file` mention behavior still works.
- Existing send/cancel behavior still works.
- No new hardcoded UI colors.
- New user-visible text is localized.

## Risks

- Slash and mention completion can conflict in the composer. Keep detection token-scoped.
- CLI feature drift can break pass-through behavior. Prefer local UI behavior for MVP.
- The suggestion popup can be fragile in WPF. Use stable dimensions and existing theme resources.
