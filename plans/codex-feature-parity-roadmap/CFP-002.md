# CFP-002: Review and Diff Pane

Priority: P1  
Status: Backlog  
Owner: AI agent  
Depends on: `CFP-001` for `/review` entry point, but the pane can be developed independently

## Goal

Add a Visual Studio-native Review/Diff pane that helps users inspect repository changes, run Codex review prompts, add inline feedback, and stage or revert changes.

## User Value

Codex review workflows are much stronger when users can inspect and comment on exact lines. Visual Studio already has strong editor and diff primitives, so this feature should integrate Codex review guidance with familiar IDE behavior.

## Functional Requirements

1. Add a Review pane or tab reachable from:
   - `/review`
   - toolbar button
   - sidebar/settings action
2. Show Git repository state for the selected working directory.
3. Support diff scopes:
   - Uncommitted changes
   - Staged changes
   - Unstaged changes
   - Branch changes against a configurable base branch
   - Last turn changes if event metadata is available
4. Show a file list with changed file status.
5. Show hunks and line-level diff content.
6. Allow opening a file at a changed line in Visual Studio.
7. Allow inline comments attached to file path and line.
8. Allow sending comments back to Codex as structured review guidance.
9. Support Git operations:
   - stage file
   - unstage file
   - revert file
   - stage hunk if feasible
   - revert hunk if feasible
10. Provide a one-click "Ask Codex to review" action.
11. Provide a one-click "Address inline comments" action.

## Architecture Plan

### Models

Add models similar to:

- `GitDiffScope`
- `GitChangedFile`
- `GitDiffHunk`
- `GitDiffLine`
- `ReviewInlineComment`
- `ReviewPaneState`

Diff line fields:

- file path
- old line number
- new line number
- line kind: context, addition, deletion, hunk header
- text
- comment count

### Services

Add `GitDiffService`:

- runs `git status --porcelain=v1`
- runs `git diff --no-ext-diff --unified=<n>`
- runs `git diff --cached --no-ext-diff`
- runs `git diff <base>...HEAD`
- parses unified diff into structured models

Add `GitReviewCommandService` only if needed:

- stages/unstages/reverts files
- validates paths remain inside repository root
- handles command failures with localized messages

### View Model

Add review state to `CodexToolWindowViewModel` or a nested `ReviewPaneViewModel`:

- `IsReviewPaneOpen`
- `SelectedDiffScope`
- `ReviewChangedFiles`
- `SelectedReviewFile`
- `SelectedReviewHunks`
- `ReviewInlineComments`
- `RefreshReviewCommand`
- `RunReviewCommand`
- `AddressReviewCommentsCommand`
- `StageFileCommand`
- `UnstageFileCommand`
- `RevertFileCommand`

Prefer a nested view model if the main view model grows too much.

### UI

Use WPF controls with Visual Studio theme resources:

- compact header with scope selector and refresh button
- file list
- diff viewer with monospace text style
- inline comment affordance
- comment editor
- action bar

Do not create nested card-heavy layouts.

## Implementation Steps

6. Add minimal service tests if a test project exists. If no test project exists, keep parsing logic deterministic and verify through build/manual tests.

9. Wire `/review` from CFP-001 to open this pane.

    - build a prompt with current diff scope
    - keep prompt concise; do not paste huge diffs blindly
    - prefer instructing Codex to inspect files when possible

    - store comments in memory for the current thread
    - include file path and line number in follow-up prompt

13. Add localization strings.
14. Build and run diff hygiene.

## Acceptance Criteria

- User can open Review pane from UI.
- `/review` opens Review pane when CFP-001 exists.
- User can view uncommitted diff grouped by file and hunk.
- User can switch staged/unstaged scope.
- User can open a changed file line in Visual Studio.
- User can add inline comments and send them to Codex.
- User can stage/unstage/revert files with confirmation before destructive revert.
- Diff parser handles added, deleted, renamed, and modified files at minimum.
- UI respects light and dark Visual Studio themes.

## Risks

- Unified diff parsing can get complex. Keep MVP to file/hunk/line parsing and defer rename similarity details.
- Hunk-level stage/revert is harder on Windows. File-level actions are required for MVP; hunk-level actions can be a follow-up.
- Destructive revert must require explicit confirmation.
